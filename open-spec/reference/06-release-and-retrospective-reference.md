# 06 发布与复盘参考

## 目标

基于测试结论和风险状态完成发布决策，明确发布步骤、监控回滚和复盘改进，保证可运维与可审计。

## 输入清单

- `06-test-cases.md`（执行结果与未关闭风险）
- `05-development-plan.md`（实施偏差与未完成项）
- 关键变更文档（00-04）

## 发布决策步骤

1. 发布范围确认：明确本次变更边界与影响系统。
2. 质量闸门确认：检查阻断缺陷、未关闭风险、兼容性影响。
3. 运行准备：确认监控项、告警阈值、值班与应急联系人。
4. 回滚预案：明确触发条件、回滚窗口、回滚步骤和数据恢复策略。
5. 发布批准：记录发布结论与责任人。

## 发布执行要点

- 升级步骤必须可复现，避免隐式人工知识。
- 关键指标需在发布后观察窗口内持续监控。
- 触发回滚条件要可量化，不用主观描述。

## 复盘步骤

1. 结果对比：对比目标指标与实际结果。
2. 偏差归因：区分需求偏差、设计偏差、实施偏差、验证偏差。
3. 改进动作：每个改进项必须有 owner 和截止日期。
4. 资产沉淀：将复盘结论回写到规范/计划/测试策略。

## 常见问题与修正

- 问题：发布说明只写“已上线”。
  修正：补齐范围、步骤、监控与回滚。
- 问题：复盘无 owner。
  修正：每条改进动作绑定 owner + deadline。
- 问题：风险只列不处理。
  修正：补齐临时控制与长期治理路径。

## 自检清单

- [ ] 发布范围与兼容性影响明确
- [ ] 回滚方案可执行且有触发条件
- [ ] 监控与告警项可落地
- [ ] 复盘动作具备 owner 和截止日期

## 产出映射

- 主文档：`07-release-retrospective.md`
- 阶段交接：`08-stage-handoff.md`
- 每日记忆：当日 `docs/open-spec/memory/YYYY-MM-DD.md`

## Input Contract（子 Agent 输入契约）

- 任务目标：完成发布决策并沉淀可执行复盘改进。
- 必读输入：`06-test-cases.md`、`05-development-plan.md`、关键变更文档。
- 上游引用：测试阶段 handoff 与未关闭风险。
- 输出约束：发布策略、监控与回滚必须可执行。

## Output Contract（子 Agent 输出契约）

- 必交文档：`07-release-retrospective.md`。
- 必含内容：发布范围、升级步骤、回滚条件、复盘动作（owner + deadline）。
- 追溯锚点：CR/REV/TC/风险项 ID。
- 交接摘要：写入 `08-stage-handoff.md` 的发布结论与后续行动。
