# 05 测试与验证参考

## 目标

将需求和实施结果转为可执行、可追溯的验证活动，确保主路径、异常路径和回归风险均被覆盖。

## 输入清单

- `01-requirements.md`（FR/NFR）
- `05-development-plan.md`（TASK 状态与实施偏差）
- `02-specification.md`（行为与异常语义）

## 设计步骤

1. 追溯建模：建立 FR -> TASK -> TC 映射。
2. 分层测试：按单元、集成、端到端分配测试责任。
3. 覆盖补齐：补主路径、失败路径、边界值、回归敏感点。
4. 数据准备：明确测试数据、前置状态、环境约束。
5. 结果判定：定义通过/失败标准与证据要求。

## 用例设计规则

- 每个 TC 只验证一个核心断言。
- 用例描述要包含前置条件、步骤、期望结果。
- 失败场景必须包含错误语义与系统恢复行为。
- 回归用例优先覆盖高变更区与历史缺陷区。

## 执行与证据

- 执行记录必须包含执行人、时间、结果、关联缺陷。
- 对失败用例需标记：是否阻断发布、临时规避方案、修复计划。
- 若实现偏差存在，测试结论需明确“已接受偏差/未接受偏差”。

## 常见问题与修正

- 问题：只有 happy path。
  修正：补齐异常、边界、并发冲突场景。
- 问题：TC 与 FR/TASK 无关联。
  修正：在用例表补齐追溯列并回填。
- 问题：测试结果只有通过/失败。
  修正：补齐证据与缺陷链接。

## 自检清单

- [ ] TC 全部可追溯到 FR 与 TASK
- [ ] 测试分层合理且有覆盖依据
- [ ] 失败与边界场景充分
- [ ] 执行记录完整且可审计

## 产出映射

- 主文档：`06-test-cases.md`
- 阶段交接：`08-stage-handoff.md`
- 每日记忆：当日 `docs/open-spec/memory/YYYY-MM-DD.md`

## Input Contract（子 Agent 输入契约）

- 任务目标：验证实现是否满足 FR/NFR 与规范约束。
- 必读输入：`01-requirements.md`、`02-specification.md`、`05-development-plan.md`。
- 上游引用：实施阶段 handoff、已知偏差与风险清单。
- 输出约束：TC 必须映射 FR 与 TASK，结果必须有证据。

## Output Contract（子 Agent 输出契约）

- 必交文档：`06-test-cases.md`。
- 必含内容：用例分层、边界与失败场景、执行记录、缺陷关联。
- 追溯锚点：TC/FR/TASK/REV ID。
- 交接摘要：写入 `08-stage-handoff.md` 的验收结论、未关闭风险、发布建议。
