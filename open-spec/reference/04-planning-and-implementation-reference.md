# 04 计划与实施参考

## 目标

把设计转成可迭代执行的任务计划，并在实施过程中持续维护 TASK 状态、偏差、阻塞与纠偏动作。

## 输入清单

- `03-technical-solution.md`
- `04-storage-design.md`（如适用）
- 需求与规范追溯关系（FR -> 方案章节）

## 计划步骤

1. 任务拆解：按可独立验证原则拆成 TASK。
2. 追溯绑定：每个 TASK 绑定 FR 与方案章节。
3. 依赖建模：标记前置依赖与并行可行项。
4. 里程碑规划：按迭代定义可验收里程碑。
5. 风险前置：高风险 TASK 优先进入早期迭代。

## 实施步骤

1. 迭代启动：确认本迭代目标 TASK 与完成定义。
2. 执行更新：每日更新 TASK 状态与完成度。
3. 偏差记录：记录延期、返工、阻塞和原因。
4. 纠偏动作：明确责任人、动作和截止时间。
5. 阶段验收：确认实现与规范一致，再进入测试阶段。

## TASK 状态标准

- 未开始：无代码或实现动作。
- 进行中：已开始但未达成完成定义。
- 已完成：通过本阶段完成定义，具备可验证证据。
- 阻塞：受外部依赖或关键问题阻断。

## 完成定义（DoD）建议

- 与关联 FR 行为一致
- 关键异常路径已处理
- 有对应测试条目或测试计划锚点
- 文档状态与实现状态同步

## 自检清单

- [ ] 每个 TASK 都有 FR 与方案映射
- [ ] 里程碑具备可验收结果
- [ ] 实施偏差有记录且有纠偏动作
- [ ] 阻塞项有负责人和预计解除时间

## 产出映射

- 主文档：`05-development-plan.md`
- 阶段交接：`08-stage-handoff.md`
- 每日记忆：`09-daily-memory-template.md` 对应当日文件

## Input Contract（子 Agent 输入契约）

- 任务目标：形成可执行迭代计划并推进实施。
- 必读输入：`03-technical-solution.md`、`04-storage-design.md`（如有）、FR 映射。
- 上游引用：方案阶段 handoff 摘要、风险与回滚限制。
- 输出约束：每个 TASK 都要有状态、完成度、偏差说明。

## Output Contract（子 Agent 输出契约）

- 必交文档：`05-development-plan.md`。
- 必含内容：任务分解、依赖、里程碑、实施进展、阻塞与纠偏动作。
- 追溯锚点：TASK/FR/REV ID。
- 交接摘要：写入 `08-stage-handoff.md` 的完成 TASK、阻塞 TASK、偏差摘要。
