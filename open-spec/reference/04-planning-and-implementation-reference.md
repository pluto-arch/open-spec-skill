# 04 计划与实施参考

## 目标

把设计转成可迭代执行的任务计划，并在实施过程中持续维护 TASK 状态、偏差、阻塞与纠偏动作。

当计划已可执行且不存在 `Blocker` 时，应自动从“计划输出”进入“实际开发实施”，不等待用户再次手动发出开始开发指令。

## 输入清单

- `03-technical-solution.md`
- `04-storage-design.md`（如适用）
- 需求与规范追溯关系（FR -> 方案章节）

## 计划步骤

1. 任务拆解：按可独立验证原则拆成 TASK。
2. 追溯绑定：每个 TASK 绑定 FR 与方案章节。
3. 依赖建模：标记前置依赖与并行可行项。
4. 里程碑规划：按迭代定义可验收里程碑。
5. 风险前置：高风险 TASK 优先进入早期迭代。

## 实施步骤

1. 自动进入实施：若 `05-development-plan.md` 已形成可执行 TASK 且不存在 `Blocker`，立即进入实际开发，不等待用户再次确认。
2. 迭代启动：确认本迭代目标 TASK 与完成定义。
3. 执行开发：按依赖顺序或并行策略推进 TASK 对应实现，而不是仅停留在计划层。
4. 执行更新：持续更新 TASK 状态与完成度。
5. 偏差记录：记录延期、返工、阻塞和原因。
6. 纠偏动作：明确责任人、动作和截止时间。
7. 阶段验收：确认实现与规范一致，再进入测试阶段。

## 阶段定向进入规则

当用户显式要求“从实施阶段开始”或“开始进入开发实施”时：

1. 优先读取现有 `05-development-plan.md`。
2. 若 05 已存在且任务可执行，直接进入 TASK 实施。
3. 若 05 不存在但 `03-technical-solution.md` 与 `04-storage-design.md` 已齐备，则先生成 05，再自动进入实施。
4. 若实施所需前置文档缺失，则仅返回缺失项与阻塞原因，不得伪造实施进展。

## TASK 状态标准

- 未开始：无代码或实现动作。
- 进行中：已开始但未达成完成定义。
- 已完成：通过本阶段完成定义，具备可验证证据。
- 阻塞：受外部依赖或关键问题阻断。

## 完成定义（DoD）建议

- 与关联 FR 行为一致
- 关键异常路径已处理
- 有对应测试条目或测试计划锚点
- 文档状态与实现状态同步

## 自检清单

- [ ] 每个 TASK 都有 FR 与方案映射
- [ ] 里程碑具备可验收结果
- [ ] 实施偏差有记录且有纠偏动作
- [ ] 阻塞项有负责人和预计解除时间

## 产出映射

- 主文档：`05-development-plan.md`
- 阶段交接：`08-stage-handoff.md`
- 每日记忆：`09-daily-memory-template.md` 对应当日文件

## Input Contract（子 Agent 输入契约）

- 任务目标：形成可执行迭代计划并推进实施。
- 必读输入：`03-technical-solution.md`、`04-storage-design.md`（如有）、FR 映射。
- 上游引用：方案阶段 handoff 摘要、风险与回滚限制。
- 输出约束：每个 TASK 都要有状态、完成度、偏差说明；若计划已具备实施条件，必须继续推进实际开发而不是止步于计划文档。

## Output Contract（子 Agent 输出契约）

- 必交文档：`05-development-plan.md`。
- 必含内容：任务分解、依赖、里程碑、实施进展、阻塞与纠偏动作。
- 追溯锚点：TASK/FR/REV ID。
- 交接摘要：写入 `08-stage-handoff.md` 的完成 TASK、阻塞 TASK、偏差摘要。
- 自动闭环要求：如果当前已满足测试阶段入口条件，向 Workflow Lead 明确返回“可进入测试阶段”；如果未满足，则明确返回剩余 TASK 与阻塞原因。
