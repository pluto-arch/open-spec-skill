# 01 需求分析参考

## 目标

把输入目标转成可测试、可追溯、可评审的需求条目（FR/NFR），并明确边界与风险。

## 输入清单

- 业务目标与成功指标
- 关键干系人与使用场景
- 时间、成本、合规、性能等约束
- 现有系统边界与依赖方

## 分析步骤

1. 问题澄清：明确“要解决什么”“不解决什么”。
2. 用户场景拆分：按主路径、异常路径、边界路径拆场景。
3. 能力抽取：把场景转成能力点，再转 FR。
4. 非功能收敛：提取性能、安全、可用性、可观测性要求（NFR）。
5. 验收前置：每条 FR 必须有可验证验收标准。
6. 风险登记：识别不确定项并写入信息缺口清单。

## FR 编写规范

- 一条 FR 只描述一个可交付能力。
- 使用“触发条件 + 系统行为 + 可验证结果”的结构。
- 必须能被至少一个测试用例覆盖。

推荐句式：

- 当 <触发条件> 发生时，系统应 <行为>，并且 <可验证结果>。

## NFR 编写规范

- 约束必须可量化。
- 至少包含度量方式与阈值。

示例：

- NFR-001 性能：95 分位响应时间 <= 300ms，统计窗口 15 分钟。

## 自检清单

- [ ] FR 全部具备唯一 ID 且可测试
- [ ] In Scope / Out of Scope 明确
- [ ] NFR 量化且可观测
- [ ] 每条风险均有缓解策略或待决计划
- [ ] 信息缺口集中记录，无散落待办

## 产出映射

- 主文档：`01-requirements.md`
- 变更场景补充：`00-change-request.md`
- 阶段交接：`08-stage-handoff.md`

## Input Contract（子 Agent 输入契约）

- 任务目标：本阶段要回答的问题与完成边界。
- 必读输入：业务目标、约束、已知风险、相关历史记忆。
- 上游引用：变更请求（如有）与关键上下文摘要。
- 输出约束：FR/NFR 必须唯一编号并可测试。

## Output Contract（子 Agent 输出契约）

- 必交文档：`01-requirements.md`（变更场景补 `00-change-request.md`）。
- 必含内容：范围、FR/NFR、验收标准、信息缺口、风险假设。
- 追溯锚点：FR/NFR/CR/REV ID。
- 交接摘要：写入 `08-stage-handoff.md` 的阶段结论、风险、下阶段输入。
