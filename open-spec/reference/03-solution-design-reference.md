# 03 方案设计参考

## 目标

把规范落成可实施方案，明确架构、模块边界、技术决策、风险与回滚。

## 输入清单

- `02-specification.md`
- 关键 FR/NFR 列表
- 技术栈与运行环境约束

## 设计步骤

1. 方案范围：声明影响模块与非影响模块。
2. 模块分解：定义模块职责、接口、依赖方向。
3. 关键流程：给出主流程与异常流程步骤。
4. ADR 记录：对关键取舍写“方案-备选-决策理由”。
5. 存储设计：涉及持久化时补齐模型、索引、迁移、回滚。
6. 风险闭环：为每个高风险点定义验证动作。
7. 缺口分级：将架构边界、技术取舍、运行环境限制中的未决信息做 `Blocker/Assumption/Nice-to-know` 分类。
8. 用户补齐：若关键 ADR、依赖约束、发布回滚策略存在 `Blocker`，停止在方案阶段并向用户提问。

## 设计原则

- 低耦合高内聚：避免跨模块隐式依赖。
- 可演进：保留扩展点，避免一次性锁死。
- 可回滚：发布失败可在可接受窗口内恢复。
- 可观测：关键链路可监控、可告警、可排查。

## 存储设计准则

- 字段定义必须包含类型、约束、语义。
- 索引只为查询路径服务，避免过度索引。
- 迁移脚本必须包含前向与回退策略。

## 自检清单

- [ ] 方案章节和 FR 已完成映射
- [ ] 关键决策具备 ADR 记录
- [ ] 风险与回滚可执行
- [ ] 存储方案清晰或明确 N/A 原因
- [ ] 所有方案级 `Blocker` 已整理为待确认问题，未确认前阶段状态为 `NEEDS_USER_INPUT`

## 产出映射

- 主文档：`03-technical-solution.md`
- 存储文档：`04-storage-design.md`
- 阶段交接：`08-stage-handoff.md`

## Input Contract（子 Agent 输入契约）

- 任务目标：把规范转成可执行技术方案。
- 必读输入：`02-specification.md`、FR/NFR 列表、技术栈边界。
- 上游引用：规范阶段 handoff 摘要与关键风险。
- 输出约束：方案必须给出风险、回滚与可观测性设计。

## Output Contract（子 Agent 输出契约）

- 必交文档：`03-technical-solution.md`，涉及存储时补 `04-storage-design.md`。
- 必含内容：模块边界、关键流程、ADR、风险与回滚、存储策略/N/A。
- 追溯锚点：FR/NFR/ADR/REV ID。
- 交接摘要：写入 `08-stage-handoff.md` 的方案结论与下阶段输入。
- 若存在 `Blocker`：门禁状态必须返回 `NEEDS_USER_INPUT`，并附“待用户回答问题包（问题、原因、影响决策）”。
