# 02 规范制定参考

## 目标

把需求条目转成明确、无歧义、可实现的规范定义，覆盖行为、数据、接口、异常与兼容策略。

## 输入清单

- 已评审需求：FR/NFR
- 关键业务规则与约束
- 外部接口或依赖协议

## 制定步骤

1. 建立追溯表：先列 FR 到规范章节的映射。
2. 行为规范：输入、输出、前置、后置、状态变化。
3. 数据规范：实体字段、类型、约束、不变量。
4. 接口规范：请求/响应、错误码、幂等性、版本策略。
5. 异常规范：失败类型、回退行为、重试策略。
6. 验收对齐：把 AC 与 FR 一一对应。

## 规范质量标准

- 无冲突：同一业务规则不能出现多种解释。
- 无遗漏：主路径、异常路径、边界路径都有定义。
- 可验证：每条关键规则可被测试用例落地。
- 可兼容：明确向后兼容策略与破坏性变更条件。

## 常见问题与修正

- 问题：只有“应该支持”这种模糊描述。
  修正：补充可观测结果和边界条件。
- 问题：只写成功路径。
  修正：补齐错误语义、超时、并发冲突策略。
- 问题：接口字段未定义约束。
  修正：补齐必填、默认值、取值范围、校验规则。

## 自检清单

- [ ] 所有主要章节都能追溯到 FR
- [ ] 异常与边界语义已定义
- [ ] 兼容与版本策略已写明
- [ ] 验收标准可直接转测试用例

- 输出约束：行为、数据、接口、异常、兼容策略必须完整。

## Output Contract（子 Agent 输出契约）

- 必交文档：`02-specification.md`。
- 必含内容：行为规范、数据规范、接口规范、异常语义、验收对应。

- 交接摘要：写入 `08-stage-handoff.md` 的规范覆盖结论与未决项。
